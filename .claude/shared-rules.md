# Team-wide Claude Code rules (LetUsDoIT)

These rules apply to every LetUsDoIT repo. **Do not edit this file per-repo** — child repos get a copy via `sync-claude.sh`. Update the canonical copy in the base repo (`~/repos/LetUsDoIT/LetUsDoIT/.claude/shared-rules.md`) and run `sync-claude-all.sh` to propagate to every child repo and push.

## Side-task skills: spawn a subagent

When the user asks mid-conversation for a transactional side task that has its own skill — creating a GitHub issue, PR, branch, newsletter draft, etc. — **do not invoke the skill in-conversation**. Instead spawn a general-purpose subagent via the `Agent` tool and pass it a self-contained brief that instructs it to use the relevant skill.

**Why:** Skills loaded via the in-conversation `Skill` tool inherit the full conversation context. In a long chat this can exceed the 200K-token standard context and force a slow, expensive 1M-context escalation — or simply fail if 1M isn't enabled on the account.

**How the subagent keeps skill discipline:** Skills live in the harness (the local `.claude/skills/` directory), not the conversation. A fresh subagent still has access to the exact same skill files and will follow them in full — dup-check via `gh issue list`, set type via GraphQL, add to project, set current sprint, link to an epic if relevant, etc. Nothing is lost by moving out of the main conversation.

**The brief must be self-contained.** The subagent will not see the main conversation, so include everything it needs:
- Org / repo
- Title
- Full issue body (or PR body / branch details)
- Priority (P0/P1/P2)
- Any parent epic or related issues
- Relevant code references if applicable

**Exception:** if the conversation is short (early turns, small context), running the skill in-conversation is fine — spawning overhead isn't worth it.

## Updating these rules

1. Edit this file in the LetUsDoIT base repo
2. Run `./sync-claude-all.sh` from the base repo
3. Teammates pick up the change on their next `git pull` in any affected child repo
